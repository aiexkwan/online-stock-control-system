# Thinking Mode Command

## 用法
`/think [主題/問題]` 或 `/think`

## 執行流程
1. **啟動思維工具**
   - Ultrathink - 深度思考模式
   - Sequential-thinking - 邏輯推理
   - Task - 平行處理多個思考角度

2. **閱讀基礎守則**
   - 先閱讀 `docs/general_rules.md`
   - 了解系統設計原則和限制

3. **思考方向**
   - 根據用戶提供的提示進行深度思考
   - 必要時使用 Brave Search MCP 搜尋資訊
   - 多角度分析問題

4. **角色建議**
   - 閱讀 docs\role_play 內的角色文檔
   - 主要角色: 📊 Analyzer（深度分析）
   - 思考組合: 戰略、技術、業務、創新等不同組合

## 核心思考原則
### 🎯 KISS 原則檢查
```yaml
評估標準:
  必要性: 這個功能/結構是否真的必要？
  複雜度: 是否過於複雜？有更簡單的方案嗎？
  維護性: 未來維護會否困難？
  效益比: 投入產出比是否合理？
```

### 🚫 避免過度優化
- **過度抽象**：是否為了未來可能性而過度設計？
- **過早優化**：是否在沒有性能問題時就優化？
- **過度工程**：是否用大砲打蚊子？

### ✅ 簡單直接原則
- **直接解決**：最簡單的解決方案是什麼？
- **漸進改進**：可否先簡單實現，後續優化？
- **實用主義**：解決實際問題，而非理論完美

## 思考框架
### 1. 問題定義
```
- 真正的問題是什麼？
- 誰會受影響？
- 影響範圍多大？
```

### 2. 現狀分析
```
- 現有方案的優缺點
- 為何需要改變？
- 改變的成本與收益
```

### 3. 方案評估
```
- 列出所有可能方案
- 評估每個方案的複雜度
- 選擇最簡單有效的
```

### 4. 風險考慮
```
- 實施風險
- 維護風險
- 技術債務
```

## 常見思考主題
- 架構設計是否過度複雜？
- 功能實現是否有更簡單方法？
- 技術選型是否合適？
- 流程是否可以簡化？
- 代碼是否需要重構？
